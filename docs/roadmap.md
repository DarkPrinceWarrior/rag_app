# Корпоративный инструмент перевода и анализа документации (EN→RU)

## 0. Фиксация скоупа (что строим, что выкидываем)

Из ТЗ заказчика убираем десктопное приложение (противоречит же его собственному п. 4.1, где способ доступа — «веб-версия и виджет»). Итоговый продукт:

1. **Веб-приложение** — загрузка документов, перевод с сохранением структуры, side-by-side просмотр оригинал/перевод, редактирование, чат с документом (RAG), библиотека, экспорт.
2. **Браузерное расширение (MV3)** — перевод выделенного текста, перевод текущей страницы, быстрая загрузка файла, история, переход в веб-версию.
3. **Бэкенд-платформа** — пайплайн парсинг → OCR → перевод → реконструкция документа; LLM-инференс полностью on-premise (внешние сервисы перевода в компании запрещены — это жёсткое ограничение из ТЗ, оно диктует весь стек: **только open-weight модели, ничего не уходит наружу**).

Ключевые нефункциональные требования:
- Документ 100+ страниц → перевод за минуты, не часы (асинхронный пайплайн, постраничный параллелизм).
- Сканы с печатями/штампами/низким разрешением → OCR обязателен.
- Таблицы и спецификации: цифры и единицы измерения не искажаются (отдельный контур валидации).
- Результат — редактируемый DOCX/XLSX, а не картинка.

## 3. Пайплайн обработки документа (ядро системы)

### 3.1. Ingestion и маршрутизация

```
upload → virus scan (ClamAV) → определение типа →
  PDF:   pdffonts → есть текстовый слой? 
         ├─ да  → text-PDF ветка
         └─ нет → scan-PDF ветка (OCR)
  DOCX/XLSX/PPTX → OOXML ветка
  JPG/PNG        → OCR ветка
  TXT/HTML(web)  → plain ветка
```

Детект текстового слоя — `pdffonts`/pypdfium2 за миллисекунды, без запуска тяжёлых моделей. Смешанные PDF (часть страниц текст, часть скан) маршрутизируются постранично.

### 3.2. Парсинг → структурное представление

Любой вход приводится к **единому промежуточному формату** (JSON, идейно как DoclingDocument / MinerU content_list): дерево блоков `{type: heading|paragraph|table|figure|formula|stamp, bbox, page, text, children}`.

- **MinerU 2.5** — основной пайплайн: layout-анализ отдельно от распознавания контента, чистка колонтитулов, таблицы в HTML, формулы в LaTeX. Бенчмаркается в топе OmniDocBench (CVPR 2025 benchmark, лидерборд обновляется до сих пор — туда же входят PaddleOCR-VL-1.5, GLM-OCR, MinerU2.5).
- **PaddleOCR-VL (0.9B)** — OCR-движок для худших сканов: печати, штампы, перекос, низкое разрешение. Ультракомпактная VLM (NaViT-энкодер + ERNIE-0.3B), 109 языков, в обзорах 2026 её главное достоинство — именно устойчивость к «грязным» реальным документам, а не лабораторным примерам.
- Эскалация: если средний confidence страницы ниже порога — страница рендерится в изображение и прогоняется через **Qwen3-VL** (уже стоит в vLLM) как fallback «умного OCR».

Этот же структурный JSON — источник чанков для RAG (раздел 5), т.е. парсим **один раз**.

### 3.3. Перевод: три ветки по типу источника

**A. PDF (текст и скан) → переведённый PDF + DOCX**
**BabelDOC** (funstory-ai) через обвязку **PDFMathTranslate-next**: layout-детекция (DocLayout-YOLO), извлечение текста с координатами, перевод через наш vLLM (OpenAI-совместимый endpoint — поддерживается нативно), реконструкция PDF с исходной вёрсткой, включая многострочные формулы и таблицы; есть режим bilingual (оригинал и перевод чередующимися страницами) — это закрывает требование «сравнение оригинала и перевода» прямо в экспорте. Параллельно из переведённого структурного JSON собираем редактируемый DOCX (python-docx) — это требование экспорта из ТЗ.

**B. DOCX / XLSX / PPTX → тот же формат, та же вёрстка**
Никакой конвертации в PDF. Идём по OOXML:
- DOCX: обход `paragraph.runs`, склейка run'ов одного абзаца в сегмент (иначе LLM получит рваные куски из-за дробления Word'ом), перевод сегмента, обратное распределение по run'ам с сохранением форматирования первого run'а как доминирующего.
- XLSX: openpyxl, переводим **только** ячейки со строковым типом; числа, формулы, форматы не трогаем по построению — это автоматически гарантирует требование «цифры не съезжают».
- PPTX: python-pptx, текстовые фреймы + заметки; авто-уменьшение кегля при перерастании текстбокса (русский текст в среднем на 15–20% длиннее английского).

**C. Веб-страницы**
Расширение собирает текстовые узлы DOM (TreeWalker), батчирует по ~2–4K токенов с сохранением идентификаторов узлов, шлёт на API, заменяет текст in-place (паттерн Immersive Translate). Никакой выгрузки HTML на сервер целиком — переводим только текст, разметка остаётся в браузере.

### 3.4. Качество перевода: терминология и валидация

1. **Глоссарий** — таблица в Postgres (EN-термин → утверждённый RU-термин, домен: строительство/договоры/нефтегаз). Перед переводом сегмента релевантные термины инжектятся в промпт (constrained terminology); BabelDOC сам умеет document-level terminology extraction — используем.
2. **Контекстное окно**: переводим не сегмент в вакууме, а с 1–2 соседними абзацами и заголовком раздела — резко снижает рассогласование терминов между страницами.
3. **Числовая валидация** (post-check, дёшево и обязательно): регэксп-экстракция всех чисел/единиц из source и target сегмента; расхождение → автоматический ре-перевод с указанием ошибки, после второго фейла → флаг «требует проверки» в UI. Это прямой ответ на требование ТЗ про спецификации.
4. **Оценка качества**: офлайн-харнесс на **COMET / MetricX-24** + выборочная разметка инженерами; A/B Qwen3-32B vs Hunyuan-MT-7B на корпусе реальных документов компании до фиксации дефолтной модели.

### 3.5. Производительность (цель «часы → минуты»)

- Перевод страниц — параллельно: документ 100 стр ≈ 300–500 сегментов → батчи по N конкурентных запросов в vLLM (continuous batching там нативный).
- Прикидка: Qwen3-32B-AWQ на A100-40 даёт ~1–2K tok/s в батче на реплику → две реплики ~2–4K tok/s; 100-страничный документ ≈ 60–80K токенов выхода → **2–5 минут** перевода + 1–2 минуты парсинга. Влезаем в «минуты».
- Прогресс — SSE-стрим в UI постранично (пользователь видит, как документ переводится, может начать читать первые страницы).

## 4. LLM-инфраструктура

### 4.1. Модели (все open-weight, всё офлайн)

| Роль | Модель | Лицензия | VRAM (квант) |
|---|---|---|---|
| Перевод (качество) + RAG-чат | Qwen3-32B, AWQ INT4 (FP8 на A100/Ampere недоступен — compute 8.0) | Apache 2.0 | ~20 GB весов → 1×A100-40 |
| Перевод (скорость, виджет) | Hunyuan-MT-7B | Tencent license (проверить юристами) | ~16 GB FP16 / 8 GB INT4 |
| OCR-fallback / чертежи | Qwen3-VL (8B/32B) | Apache 2.0 | 16–40 GB |
| Эмбеддинги | BGE-M3 → Qwen3-Embedding-4B | MIT / Apache 2.0 | 2–10 GB |
| Reranker | Qwen3-Reranker-0.6B | Apache 2.0 | ~2 GB |

Почему Qwen3, а не Llama 4 / Gemma 4: (а) Apache 2.0 без ограничений Llama-лицензии; (б) в обзорах 2026 Qwen3 — консенсусный «безопасный энтерпрайз-выбор» для multilingual; (в) одна модель закрывает и перевод, и чат, и structured output → один деплой, один прогрев, общий KV-кэш.

Почему Hunyuan-MT-7B как второй контур: победа в WMT2025 (30/31 направлений) при 7B — лучшее качество/латентность для интерактивного перевода выделенного текста; направление en→ru входит в её 33 языка.

### 4.2. Сервинг

- **vLLM** ≥ v0.10: OpenAI-совместимый API, prefix caching, continuous batching, AWQ-Marlin. Все воркеры ходят в него по `OPENAI_BASE_URL` — это даёт бесплатную замену движка (SGLang при росте RAG-нагрузки: его RadixAttention даёт документированные 30–40% экономии GPU на shared-prefix трафике, а чат с документом — это именно он) и бесплатную замену модели.
- Эмбеддинги/reranker — **Text Embeddings Inference** (HF TEI) отдельным контейнером.

### 4.3. Раскладка под фактическое железо (Proxmox, хост `zeta`, LXC CT 135: 6×A100-SXM4-40GB)

Финальная диагностика (2026-06-11, проверено на хосте): **NVLink физически отсутствует** — SXM4-модули на PCIe-карьере, без HGX/NVSwitch (ноль линков в `nvlink -c`, пустой `dmesg | grep nvlink`, ни одного NV# в `topo -m`, NVSwitch в `lspci` нет). Связь — только PCIe Gen4 ×16 (~31.5 GB/s/напр. против 600 GB/s у NVLink3), P2P недоступен → `NCCL_P2P_DISABLE=1` — **постоянная корректная настройка**, NCCL-коллективы идут через host-RAM. `nvidia-fabricmanager` не ставить — управлять нечем. Два NUMA-острова: GPU 0–2 (node0) и GPU 3–5 (node1, пара 4↔5 = PIX); A↔B = SYS (UPI).

**Следствие: tensor parallelism исключён архитектурно и навсегда** (не лечится софтом). Вместо TP — **data parallelism**: Qwen3-32B-AWQ (~20 GB весов) целиком влезает на одну A100-40, межкартный трафик отсутствует в принципе — отсутствие NVLink для этой схемы безразлично.

Раскладка (whisperx на GPU0 остановлен 2026-06-11 — проверить, что сервис disabled, а не stopped, иначе вернётся после ребута):

| GPU | NUMA | Сервис | Заметки |
|---|---|---|---|
| 0 | node0 | Qwen3-32B-AWQ — реплика A | перевод + чат |
| 1 | node0 | Qwen3-32B-AWQ — реплика B | за общим роутером (LiteLLM proxy / nginx least_conn) |
| 2 | node0 | Qwen3-VL-8B | OCR-fallback, чертежи |
| 3 | node1 | Hunyuan-MT-7B | быстрый контур виджета, изолирован от batch-нагрузки |
| 4 | node1 | TEI (BGE-M3 + Qwen3-Reranker) + PaddleOCR-VL | мелкие burst-модели |
| 5 | node1 | Резерв: LoRA-дообучение, A/B, стейджинг | при росте → третья реплика 32B; обучение — по рецептам Tier 1/2 runbook'а |

Правила эксплуатации:
- KV-кэш: `--kv-cache-dtype fp8_e5m2` (формат хранения, на Ampere работает) → ~100K+ токенов кэша на реплику при ~16 GB свободной VRAM.
- NUMA-пиннинг обязателен (H2D/D2H-копии не должны ходить через UPI): процессы GPU0–2 → `numactl --cpunodebind=0 --membind=0`, GPU3–5 → node1. `NCCL_P2P_DISABLE=1 NCCL_IB_DISABLE=1` — в окружении постоянно.
- Если в будущем понадобится модель >40 GB: **PP, а не TP**, через границу островов (PP передаёт только активации на границе стейджа — нечувствителен к host-staging); схема TP=2 внутри острова (пара 4↔5 = PIX) × PP=2 между островами — и только для compute-bound задач.
- **ECC оставить Enabled.** Рекомендация выключить ради +1–2 GB/карту годится для research; для прод-сервиса, переводящего договоры и спецификации, тихий bit flip = неотлавливаемое искажение цифр — ровно то, против чего в ТЗ отдельное требование.
- Диск 492 GB мал: веса моделей (~100 GB) + MinIO + Postgres → запросить отдельный том 1–2 TB под `/models` и данные MinIO.
- CPU: контейнер видит 64 из 128 потоков хоста — попросить `pct set 135 -cores 96`, CPU-стадии MinerU и ARQ-воркеры утилизируют.

### 4.4. Специфика деплоя в LXC (не KVM!)

CT 135 — LXC-контейнер: драйвер NVIDIA живёт на хосте `zeta` (580.95.05), внутрь прокинуты `/dev/nvidia*`, userspace-библиотеки в контейнере обязаны **точно совпадать с версией драйвера хоста** (обновление драйвера на хосте = синхронное обновление либ в CT).

- **Основной путь**: GPU-сервисы (vLLM ×3, TEI, OCR) — напрямую systemd-юниты + uv внутри CT, без Docker-прослойки. Меньше слоёв — меньше сюрпризов с драйвером.
- Stateless-инфраструктура (Postgres, Redis, MinIO, FastAPI, фронт) — docker-compose внутри CT: требует `features: nesting=1` на контейнере и nvidia-container-toolkit с `no-cgroups = true`, если какой-то GPU-сервис всё же пойдёт в Docker (cgroup-делегирование устройств в nested-контейнерах не работает).
- Перепроверка железа после любых изменений платформы — по чек-листу Приложения B runbook'а (`nvidia-smi nvlink -c`, `topo -m`, `lspci | grep -i nvswitch`).

## 5. RAG: «чат с документом»

Тренд 2026 — **не** тащить LangChain: тонкая собственная оркестрация (или Pydantic AI как лёгкая прослойка), полный контроль промптов, трейсинг в Langfuse.

1. **Чанкинг по структуре**, не по символам: чанк = раздел/подраздел/таблица из структурного JSON парсера (метаданные: страница, bbox, заголовок-путь «4 → 4.3 → таблица 2»). Таблицы — отдельные чанки с markdown-представлением + заголовок таблицы.
2. **Гибридный поиск**: dense (BGE-M3) + sparse (BM25 — `tsvector` в том же Postgres или sparse-векторы BGE-M3) → RRF-слияние → **Qwen3-Reranker** top-20 → top-5. Гибрид критичен именно для технички: артикулы, номера ГОСТов, аббревиатуры dense-поиском ловятся плохо.
3. **Двуязычный индекс**: эмбеддим и оригинал (EN), и перевод (RU) с перекрёстной ссылкой — вопрос на русском находит и английский исходник (BGE-M3/Qwen3-Embedding кросс-язычны по построению).
4. **Цитаты обязательны**: ответ содержит ссылки на чанки → UI подсвечивает место в документе (bbox есть из парсера). Это главная защита от галлюцинаций в договорной тематике.
5. **История и контекст** (п. 4.5 ТЗ): сессии чата в Postgres; при новом вопросе — последние N реплик + суммаризация старых в системный промпт.
6. Спец-интенты из ТЗ («вытащи спецификации в таблицу») — structured output (JSON-режим vLLM) → рендер таблицы → экспорт XLSX.
7. **Адаптивный agentic-уровень** (тренд 2026 — не «agentic везде», а роутинг по сложности). Быстрый классификатор (промпт к Qwen3 со structured output) делит запросы:
   - *single-hop* («найди раздел про ответственность», «что сказано про испытания») → обычный hybrid+rerank, ответ за 2–3 с;
   - *multi-hop / экстракция* («вытащи все спецификации в таблицу», «сравни этот пункт с тем, что обсуждали вчера») → агентный цикл: retrieval как tool call, инструменты `search_chunks`, `get_section(id)`, `get_tables(doc)`, `get_chat_history`; модель итеративно ищет, оценивает достаточность и дособирает контекст.
   - **Стоп-условия — обязательная часть дизайна** (главный провал agentic RAG — незавершающийся цикл): ≤5 итераций, ≤30K токенов на запрос, ≤60 с wall-clock. Реализация — тонкий tool-calling цикл руками (~150 строк, function calling в vLLM), без LangChain; LangGraph — только если понадобятся чекпоинты/human-in-the-loop. Все шаги цикла — в трейсы Langfuse.

## 9. Безопасность, лицензии, соответствие

- **Контур**: всё on-prem, egress c GPU-нод закрыт, модели заливаются офлайн-артефактами. Ни один байт документа не покидает периметр — главный аргумент проекта против запрещённых публичных переводчиков.
- **AGPL-3.0 у BabelDOC/PDFMathTranslate** — единственная юридическая мина стека. Для внутреннего использования без распространения ПО AGPL обычно приемлема (сеть-копилефт триггерится предоставлением сервиса третьим лицам; внутри одного юрлица — серая зона, у многих корпораций бан на AGPL политикой). Действия: (а) согласовать с юристами до начала; (б) изолировать BabelDOC в отдельный микросервис-контейнер с узким API — если придётся выпиливать, заменим только его (запасной путь: свой re-render на pypdfium2 + DocLayout-YOLO, дороже на ~3–4 недели); (в) не модифицировать исходники, конфигурировать снаружи.
- Остальной стек юридически чистый: Apache 2.0 (Qwen3, vLLM, MinerU-движки), MIT (BGE-M3, FastAPI, WXT, shadcn), PostgreSQL/BSD.
- RBAC: документ принадлежит пользователю; папки/проекты с шарингом; pgvector-поиск всегда фильтруется по правам в том же SQL-запросе (нет отдельной vector-БД → нет рассинхрона прав).
- Аудит: кто загрузил/перевёл/экспортировал/спросил — append-only audit_log (требование для договорных документов).
- ClamAV на загрузке; лимиты размера; PDF-санитайзер (pdfcpu) против JS-вложений.

## 11. План работ (MVP → прод)

**Этап 1 — скелет (3–4 нед).** Docker-compose: Postgres+pgvector, MinIO, Redis, FastAPI, vLLM c Qwen3-32B-AWQ. Загрузка PDF → MinerU-парсинг → структурный JSON → сегментный перевод → DOCX-экспорт. Примитивный UI списка документов. *Критерий: текстовый 50-страничный PDF переводится в DOCX < 10 мин.*

**Этап 2 — вёрстка и сканы (3–4 нед).** BabelDOC-микросервис (PDF→PDF), OCR-ветка (PaddleOCR-VL), OOXML-транслятор DOCX/XLSX/PPTX, числовая валидация, глоссарий. Side-by-side вьювер с правками. *Критерий: скан договора с печатями → переведённый PDF с сохранённой вёрсткой + редактируемый DOCX.*

**Этап 3 — RAG и библиотека (3 нед).** Гибридный индекс, reranker, чат с цитатами и подсветкой, история сессий, папки/поиск по библиотеке, экспорт выжимки и истории чата.

**Этап 4 — расширение (2–3 нед).** WXT: перевод выделения, перевод страницы, popup, OIDC, корпоративная раздача.

**Этап 5 — прод (2–3 нед).** Keycloak/AD SSO, RBAC, аудит, Langfuse, нагрузочное (20 одновременных документов), бенчмарк качества COMET на корпусе компании, при необходимости — переезд translate-контура на SGLang.

Итого ~3,5–4,5 месяца одним сильным фуллстеком + поддержка DevOps на GPU-инфраструктуру.
